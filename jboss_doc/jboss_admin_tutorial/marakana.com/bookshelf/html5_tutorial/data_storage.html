7. Data Storage

7.1. Problems With The Existing Cookie-based Model

  • Invented during the early days of the internet, a cookie is a small piece of text that allow data storage for a client
  • Consist in a name/value pairs for storing data
  • Often times used to remember user preferences on a web site

  • However, cookies have several downsides:

    • They are difficult to work with. There are no real API to create, read, update or delete a cookie. This can quickly become a pain for developers

      Setting a cookie.  

      document.cookie = "name=value; path=/; expires=date";

      Note

      You could then try the following document.cookie = "name=; path=/"; thinking that it would delete the cookie previously set. But it does not delete the cookie until the browser is shut down

    • They are included in every HTTP request

      • Useless because the same data is transmitted between the client and the server
      • Not secure because they are not encrypted (Unless your whole web site use SSL)
    • They are limited in terms of size: 4KB only per cookie
    • There is a limited number of cookies per domain per browser: 20 cookies per domain per browser
    • Total number of cookies per browser: 300 cookies per browser

7.2. Hacks Prior To HTML5

  • To circumvent those limitations, many other storage solutions have been invented:

    • userData for Microsoft Internet Explorer
    • Flash cookies by Adobe
    • Gears by Google
    • dojox.storage
    Note

    The problem is that all of these solutions are either specific to a single browser or are relying on a third-party plugin like flash

7.3. New Storage Options

  • HTML5 Web storage goal is to provide a standardized and native API that works the same way on any browser

  • Two ways to store data on the client side:

    • Web Storage: Key/Value pair storage system

      • localStorage
      • sessionStorage

    • Web SQL Database: SQL Database storage system

      Warning

      On November 19th, 2010 the W3C announced that they would no longer support Web SQL.

7.3.1. Web Storage

  • Originally included in the Web Applications 1.0 specification, it now has its own specification: W3C Web Storage

    Note

    "Local Storage" or "DOM Storage" are the terms used by some browser vendors for referring to web storage. Another term used to refer to web storage is "cookies on steroids".

  • Key/Value pair data is stored on the client side like cookies except that the data is not sent to the server

  • Two types of storage: sessionStorage and localStorage

    • sessionStorage: Each window or tab has its own sessionStorage object. That means that any key/value pair created in sessionStorage for a window/tab can only be accessed from this window/tab. Data remain available until this same window/tab is closed
    • localStorage: The localStorage object is shared between window/tab for a same domain and data remain available until explicitly deleted by the user using either the web application or the browser options

  • Specification recommendation says the web storage limit per browser per origin is 5MB

    Warning

    When the storage quota is exceeded, an QUOTA_EXCEEDED_ERR exception is thrown. There is currently no way for developer to ask permission for more storage to the user, that means that you have to be careful as a developer

  • sessionStorage and localStorage both share the same API to create, read, update or delete data on the client side. They both implement the Storage interface which is the following 

    interface Storage {
  readonly attribute unsigned long length; 
  getter DOMString key(in unsigned long index); 
  getter any getItem(in DOMString key); 
  setter creator void setItem(in DOMString key, in any data); 
  deleter void removeItem(in DOMString key); 
  void clear(); 
};

    		 Number of items. For instance, localStorage.length would tell us how many items do we have in the localStorage

    		Get the key of an item with a zero based index. For instance, let us say we have one item in the localStorage with the following key/value pair: "animal"/"unicorn". Calling localStorage.key(0) would return "animal"

    		Get the value of an item identified by its key. For instance, using the previous item ("animal"/"unicorn"), calling localStorage.getItem("animal") would return "unicorn"

    		Can be either used to create or modify an item. For instance, if we need to create an item to remember the name of the user: localStorage.setItem("name","Jack")

    		Delete an item identified by its key. For instance, if we need to delete the previous item: localStorage.removeItem("name")

    		Delete all items

7.3.2. Browser Support

  • Web storage is supported on Internet Explorer 8.0+, Firefox 3.5+, Safari 4.0+, Chrome 6.0+ and Opera 10.5+

7.3.3. Web SQL Storage

  • Storing key/value pair data is pretty nice but what if you want to store complex relational data?
  • Another way to store data on the client side is to use web SQL storage

  • Web SQL storage also has its own specification: Web SQL Database

    Warning

    As previously mentioned, the W3C is no longer maintaining the Web SQL Database specification (since November 18th, 2010) meaning that developers are warned that they cannot count on it and that they use it at their own peril. However, since it is still implemented in some desktop browsers and also in mobile versions of Safari on Apple’s iOS, it is still interesting to know how to use it

  • Below is an example of how to create the database and how to perform operations on it: 

    //Creates the database (if not already created) and opens it
var db=openDatabase('contacts','1.0','my contacts app', 2 * 1024 * 1024); //

//Initialize the database
//
db.transaction(function(tx) {
  //
  tx.executeSql('CREATE TABLE IF NOT EXISTS contacts(id integer primary key autoincrement, firstname, lastname, phonenumber)');
});

//Read the contacts from the database (supposing there are some)
//
db.transaction(function(tx) {
  //
  tx.executeSql('SELECT * FROM contacts', [], function(tx, results) {
    var len=results.rows.length;
    var i;
    for(i=0; i<len; i++) {
      alert("Id: " + results.rows.item(i).id + " Firstname: " + results.rows.item(i).firstname + " Lastname: " + results.rows.item(i).lastname + " Phonenumber: " + results.rows.item(i).phonenumber); //
    }
  });
});

    		 The openDatabase method opens an existing database or creates it and then opens it. It takes the following arguments: a database name, a version number, a textual description and a size.

    		Once we get the Database object (in this case named db), we can perform SQL statements on the database. These SQL statements can only be performed in a transaction. So the transaction method can take up to three arguments and only the first one is mandatory: The first argument is the transaction callback (it contains the transaction steps) and this callback is passed a SQLTransaction object. The second one, optional, is the error callback and is called if something went wrong in the transaction callback. The last one, also optional, is the success callback and is called when everything went well in the transaction callback.

    		Inside the transaction callback, we have access on a SQLTransaction object (here called tx) on which we call the executeSql method to execute SQL statements. This method takes the following arguments: The first one contains the SQL statement (can be a SQL prepared statement with parameters represented by question marks: ?). The second one represents the array of parameters for the SQL statement. The third one, optional, is the success callback and is passed two arguments: The current SQLTransaction object (called tx in this case) and a SQLResultSet object containing the results of the executed SQL statement. The fourth and last one is the error callback, executed if there was an error during the execution of the SQL statement.

    		In the success callback, executed after the SQL statement, we have access to the SQLResultSet object. With this object we can read the results from the database, when a SELECT statement was executed. We can also know the number of rows affected by a query as well as knowing the last inserted id of a tuple.
    Note

    Regarding the size of a database, the specification recommends 5MB per database per origin per user agent.

    Note

    Opening the database with the right version number is important since a database cannot exist in multiple versions at a time on the browser.

7.3.4. Browser Support

  • Web SQL storage is currently supported in Safari 3.2+, Chrome 6.0+ and Opera 10.5+

    Warning

    Since November 18th, 2010, the W3C no longer maintain the web SQL storage specification so do not expect browser support in the future.

7.4. Quiz

  2. Explain the difference between localStorage and sessionStorage
  3. What is the web storage limit per browser per origin?
  4. Where do you have to perform your database operations?
  5. What is the maximum space allowed for a web SQL database per origin per user agent? What happens is the database use more than the size limit?
