13. JMX Administration

13.1. What is JMX?

  • Management and monitoring standard
  • Both local and remote management
  • Change settings at runtime
  • Event notification / timer
  • Dynamic class loading from XML files
  • Portable across application servers
  • Integrate with 3rd-party components
  • You can use JMX to load, initialize, change, and monitor your application and its distributed components.
  • JMX is a standard for managing and monitoring all varieties of software and hardware components from Java. In addition, JMX aims to provide integration with the large number of existing management standards, such as SNMP and WBEM.

13.2. Why JMX?

  • Custom configuration management is hard to implement and typically not reusable
  • App-server-specific solutions are proprietary and tie you to the vendor
  • JMX is reusable and open framework for both local and remote configuration management that supports runtime querying/updates, monitoring, notifications, timers, class loading, etc.

13.3. JMX Architecture

Instrumentation Layer

  • The instrumentation layer exposes the application components as one or more managed beans (MBeans)
  • Each MBean provides access to its state using public methods
  • An MBean can be any Java object that supports the interfaces and semantics of the JMX specification

Agent Layer

  • The JMX agent is the hub of the JMX framework

    • It provides remote management and application access to all of its registered MBeans

  • The agent also supports additional services, such as monitoring and dynamic loading

    • These services are also implemented and registered as MBeans so that they benefit from the framework as well
  • The core agent component is called MBean server and is defined by the interface javax.management.MBeanServer

Distributed Services Layer

  • The distributed management services layer provides the interfaces and components that remote tools use to interface with agents
  • The current specification leaves the definition of these interfaces and other functionality to the future versions of JMX, but most application servers provide JMX connectors and adaptors for web (HTML), RMI, and SNMP access

13.4. JMX on JBoss AS

  • Next to the JBoss Microcontainer, JMX is at the very core of JBoss AS.
  • Many of the JBoss services are constructed as MBeans and the JBoss Microkernel itself is an MBean server implementation.

  • The minimal configuration set only starts:

    • JBoss Microcontainer
    • JMX Kernel
    • JNDI MBean
    • Logging Mbean
  • JBoss adaptors include HTML/HTTP and RMI

13.5. JMX Console

JMX Console is a deployed web application (Distributed Services Layer) that acts as a web UI into the JBoss Microkernel (Agent Layer) and all of the deployed services (Instrumentation Layer MBeans).

The following list outlines some of the JMX Console’s capabilities:

  • Reconfigure logging: jboss.system:service=Logging,type=Log4jService
  • Get JBoss Version, Run GC, Shutdown: jboss.system: type=Server
  • Get JBoss Directory Paths: jboss.system:type=ServerConfig
  • Get JVM/Memory/Thread Info: jboss.system:type=ServerInfo
  • Query an individual servlet: jboss.web:J2EEApplication=none,J2EEServer=none,WebModule =//localhost/fortune,j2eeType=Servlet,name=fortune
  • View JNDI Tree: jboss:Service=JNDIView
  • Manage a DB connection pool: jboss.jca:name=DefaultDS,service=ManagedConnectionPool
  • Start/Stop applications: jboss.web.deployment:id=2147076203,war=fortune.war
  • Change virtual host settings: jboss.web:host=localhost,type=Host
  • View and change HTTP connector settings: jboss.web:address=/0.0.0.0,port=8080,type=Connector
  • View/Flush HTTP sessions: jboss.web:host=localhost,path=/fortune,type=Manager
  • Manage deployments: jboss.system:service=MainDeployer

13.6. Web Console

  • Like JMX Console, Web Console provides the same, but richer, view into the JBoss JMX server/services
  • Navigation is done through a Java Applet
  • Supports creation of alerts and real-time monitors (right-click)
Note

To use Web Console, your browser have to support applets.

13.7. Twiddle Tool

  • Command Line access to a [remote] JMX server (similar to web-based JMX Console)

  • Its capabilities include:

    • Getting/setting attributes on MBeans
    • Invoking operations on MBeans
    • Looking up MBeans
    • Getting server information

  • Called twiddle (for twiddling bits)

    • bin/twiddle.sh (UNIX)
    • bin/twiddle.bat (Windows)
  • Great for automation

List all deployed web applications:

./twiddle.sh query "jboss.web.deployment:*"
jboss.web.deployment:war=web-console.war,id=115013145
jboss.web.deployment:war=jbossmq-httpil.war,id=753610482
jboss.web.deployment:war=ROOT.war,id=-1586513555
jboss.web.deployment:war=jboss-ws4ee.war,id=1143514564
jboss.web.deployment:war=jmx-console.war,id=-1884883379
jboss.web.deployment:war=fortune.war,id=2147076203
jboss.web.deployment:war=invoker.war,id=-1536256770

Control the life-cycle an application:

./twiddle.sh invoke jboss.web.deployment:war=fortune.war,id=2147076203 stop
'null'
./twiddle.sh invoke jboss.web.deployment:war=fortune.war,id=2147076203 start
'null'

Get a state on an application:

./twiddle.sh get jboss.web.deployment:war=fortune.war,id=2147076203 StateString
StateString=Started

Get the current memory usage of our JBoss AS instance:

./twiddle.sh get jboss.system:type=ServerInfo TotalMemory
TotalMemory=56184832

Ask JBoss AS to run the garbage collector:

./twiddle.sh invoke jboss.system:type=Server runGarbageCollector
'null'

Shutdown JBoss AS instance:

./twiddle.sh invoke jboss.system:type=Server shutdown
Note

It’s highly recommended to quote MBean’s name when using twidlle so this way special characters won’t be interpreted by the command-line.

13.8. JBoss AS Administration Console

  • New since JBoss AS release 5.1.0
  • Also known as Embedded Jopr project
  • Available at the following URL ⇒ http://localhost:8080/admin-console
  • Default administrator credentials are admin/admin

  • Essentially, a simplified version of the other console applications

    • Less clutter → easier to work with
Note

These credentials come from the jmx-console security domain. The configuration is in ${jboss.server.home.url}/conf/props/jmx-console-users.properties. Security domains will be discussed in the Security chapter.

Figure 6. The user interface:



  • Is divided in two frames

    • The left frame list the ressources available on the server
    • The right frame is the Control Panel where you can manage the ressource selected

The Control Panel - 4 main tabs:

  • Summary: Contains the general properties and the most relevant metrics
  • Configuration: To edit or create new ressources (example: you can add a new datasource and it will generate the xxx-ds.xml for you)
  • Metrics: List all the metrics for a ressource
  • Control: When enabled, you can perform special actions related to a ressource

The possiblities in the Admin Console are the following:

  • Deploy/Undeploy applications
  • Update applications
  • Start/Stop/Restart applications
  • Add/Delete ressources
  • Manage ressources
  • See metrics

13.9. Lab: JMX Print Service

  • Deploy print-service.sar

  • Invoke its operations and modify its attributes through

    • JMX Console
    • Twiddle tool
  • Try changing its state (start/stop)

Additionally, try the following (through either JMX Console or twiddle):

  • Change the session expiry interval of one of your web applications. Hint: take a look at maxInactiveInterval of host=localhost,path=/your-web-app,type=Manager
  • Get the number of active sessions for /jmx-console web application. Access it with a different browser to see if this number changes.
  • Add another alias to Tomcat Host, and confirm that the alias gets included in the list of aliases for that host. Hint: host=myhost,type=Host
  • Change the log pattern of the web server access log for localhost, test that the change is made effective immediately, and then change it back
  • Get the SMTP server configured in the mail service
  • Get the server home directory of the running JBoss instance (server)
  • Get the name of the running JBoss instance (server)
  • Get the JVM version of the JBoss instance
  • Shutdown JBoss without using CTRL+C or the shutdown script. Start it up again.

13.10. JBoss Monitoring

  • Define monitors - XML-defined MBeans that watch for values of other MBeans' attributes
  • When monitoring threshold is reached registered listeners are notified
  • Threshold Types: numeric, string-based
  • Predefined monitors: heartbeat, memory
  • Easy to create, manage, and persist via JBoss /web-console/ application

  • Fully Configurable Listeners

    • Console Listener (STDOUT), Email Listener

    • deploy/monitoring-service.xml 

      <server>
  <mbean code="org.jboss.monitor.alerts.ConsoleAlertListener"
         name="jboss.alerts:service=ConsoleAlertListener">
    <attribute name="MessageTemplate"><![CDATA[%(MONITOR_NAME) was triggered for attribute %(ATTRIBUTE).]]></attribute>
    <attribute name="AlertName">Console Alert</attribute>
  </mbean>

  <mbean code="org.jboss.monitor.alerts.EmailAlertListener"
         name="jboss.alerts:service=EmailAlertListener">
    <depends>jboss:service=Mail</depends>
    <attribute name="MessageTemplate"><![CDATA[%(MONITOR_NAME) was triggered for attribute %(ATTRIBUTE).]]></attribute>
    <attribute name="AlertName">Email Alert</attribute>
    <attribute name="To">somebody@somewhere.org</attribute>
    <attribute name="From">noreply@nowhere.org</attribute>
    <attribute name="ReplyTo">noreply@nowhere.org</attribute>
    <attribute name="SubjectTemplate"><![CDATA[[jboss-alert] %(MONITOR_NAME)]]></attribute>
  </mbean>
  ...
</server>

    • Fields MessageTemplate and SubjectTemplate can reference any of the following placeholders:

      • %(OBSERVED_OBJECT) - the MBean ObjectName that is being monitored
      • %(MONITOR_OBJECT_NAME) - the JMX ObjectName of the monitor itself
      • %(MONITOR_NAME) - the Display Name of the Monitor
      • %(ATTRIBUTE) - the attribute name of the MBean being observed
      • %(TRIGGERED_ATTRIBUTE_VALUE) - The value that cause the alert trigger
      • %(THRESHOLD) - The value of the threshold

13.10.1. Free Memory Monitor Example

deploy/management/monitors/Free_Memory_Alert-service.xml:  

<?xml version="1.0" encoding="UTF-8"?>
<server>
  <mbean code="org.jboss.monitor.ThresholdMonitor"
         name="jboss.monitor:service=Free_Memory_Alert">
    <attribute name="MonitorName">Free Memory Alert</attribute>
    <attribute name="ObservedObject">
      jboss.system:type=ServerInfo
    </attribute>
    <attribute name="ObservedAttribute">FreeMemory</attribute>
    <attribute name="Threshold">1048576</attribute>
    <attribute name="CompareTo">1</attribute>
    <attribute name="Period">1000</attribute>
    <attribute name="Enabled">true</attribute>
    <depends-list optional-attribute-name="AlertListeners">
      <depends-list-element>
        jboss.alerts:service=ConsoleAlertListener
      </depends-list-element>
    </depends-list>
  </mbean>
</server>

13.10.2. Create a monitor using Web Console

You’ve just seen an example of how can you create a monitor manually. Another way to achieve the same result is to use Web Console.

  • Open Web Console ⇒ http://localhost:8080/web-console/
  • Expand the System tree ⇒ JMX MBeans tree until you find the bean attribute you want to monitor (located on the left pane)
  • Right-click on the attribute ⇒ create monitor
  • Then you specify the monitor:

This will create a MonitorName-service.xml under deploy/management directory. If the name of your monitor contains spaces, this is replaced by underscores.

Note

Once a monitor has been triggered, it send an alert and remains in that state until you clear it. Since a monitor is a MBean you can invoke its clearAlert method.

13.11. Snapshot and Web Console

  • A Snapshot record changes of a given MBean attribute within a defined time interval

How to create a Snapshot?

  • Open Web Console ⇒ http://localhost:8080/web-console/
  • Expand the System tree ⇒ JMX MBeans tree until you find the bean attribute you want (located on the left pane)
  • Right-click on the attribute ⇒ create snapshot

Once create it will be available under Monitoring ⇒ Snapshots

From there you can:

  • Graph the results
  • View the dataset

13.12. Monitoring with JConsole

JConsole is a GUI application provided by JDK as of version 5

  • Detect low memory
  • Enable/disable GC and class loading verbose tracing
  • Detect deadlocks
  • Control the log level of any loggers in an application
  • Access OS resources—Sun’s platform extension
  • Manage an application’s Managed Beans (MBeans)

To use JConsole with JBoss:

  • Local connection

    • In run.conf for Linux: 

      JAVA_OPTS="$JAVA_OPTS -Dcom.sun.management.jmxremote"

    • In run.bat for Windows: 

      JAVA_OPTS="%JAVA_OPTS% -Dcom.sun.management.jmxremote"

  • Remote connection

    • In run.conf for Linux: 

      JAVA_OPTS="$JAVA_OPTS -Dcom.sun.management.jmxremote.port=12345"
JAVA_OPTS="$JAVA_OPTS -Dcom.sun.management.jmxremote.authenticate=false" 
JAVA_OPTS="$JAVA_OPTS -Dcom.sun.management.jmxremote.ssl=false"

      		By default, security has to be configured. These lines are useful if you want to disable security and ssl transport.

    • In run.bat for Windows: 

      JAVA_OPTS="%JAVA_OPTS% -Dcom.sun.management.jmxremote.port=12345"
JAVA_OPTS="%JAVA_OPTS% -Dcom.sun.management.jmxremote.authenticate=false" 
JAVA_OPTS="%JAVA_OPTS% -Dcom.sun.management.jmxremote.ssl=false"

      		By default, security has to be configured. These lines are useful if you want to disable security and ssl transport.

To launch JConsole just run jconsole

13.13. Scheduling on JBoss

  • JMX includes a mechanism for scheduling, it is provided by javax.management.timer.TimerMBean

  • JBoss’s variation of the JMX timer service : org.jboss.varia.scheduler.Scheduler

    • Invoke a callback on a instance of a class that implements org.jboss.varia.scheduler.Schedulable OR
    • Invoke a callback on a method of a MBean
  • Schedulers are configured in deploy/scheduler-service.xml

Here is an example of how you would run periodically the garbage collector (this is just to demonstrate, doing this is an expensive and non-reliable operation)

<server>
  ...
  <mbean code="org.jboss.varia.scheduler.Scheduler"
        name="jboss.scheduler:service=Scheduler,name=SchedulableMBeanExample">
    <attribute name="StartAtStartup">true</attribute> <!--  -->
    <attribute name="SchedulableMBean">jboss.system:type=Server</attribute> <!--  -->
    <attribute name="SchedulableMBeanMethod">runGarbageCollector()</attribute> <!--  -->
    <attribute name="InitialStartDate">NOW</attribute> <!--  -->
    <attribute name="SchedulePeriod">10000</attribute> <!--  -->
    <attribute name="InitialRepetitions">10</attribute> <!--  -->
    <attribute name="FixedRate">true</attribute> <!--  -->
  </mbean>
  ...
</server>

 StartAtStartup: If true, the scheduler will be automatically started when JBoss start. If false, an explicit startSchedule operation must be invoked on the Scheduler to begin.

SchedulableMBean: Fully qualified JMX name of the MBean to be called

SchedulableMBeanMethod: Method to be called on the schedulable MBean. It can optionally be followed by an opening bracket, a comma separated list of parameter keywords, and a closing bracket. The supported parameter keywords include:

  • NOTIFICATION which will be replaced by the timers notification instance (javax.management.Notification)
  • DATE which will be replaced by the date of the notification call (java.util.Date)
  • REPETITIONS which will be replaced by the number of remaining repetitions (long)
  • SCHEDULER_NAME which will be replaced by the ObjectName of the Scheduler
  • Any fully qualified class name which the Scheduler will set to null.

InitialStartDate: Date when the initial call is scheduled.

  • NOW: current time plus one seconds
  • number representing the milliseconds since 1/1/1970
  • Date as String formatted with this default pattern "M/d/yy h:mm a"

SchedulePeriod: Interval between calls in milliseconds.

InitialRepetitions: number of times the scheduler will invoke the scheduled MBean

FixedRate: The default scheduling to use, fixed-rate or fixed-delay (false, default)

13.14. Lab: Monitoring

  • Deploy Free_Memory_Alert-service.xml and JMX_Console_Sessions_Alert-service.xml in deploy/management/monitors
  • Enable Email Listener (depends on Mail service) in deploy/monitoring-service.xml
  • Create your own monitors either through the Web Console or manually

For example, create the following monitors:

  • Alert when the maximum processing time for the /status (Status Servlet) goes over 50ms. Watch for maxTime in jboss.web:J2EEApplication=none,J2EEServer=none,WebModule=//localhost/,j2eeType= Servlet,name=Status Servlet
  • Alert when the there are more than 5 concurrent requests going over the HTTP connector. Watch for currentThreadCount in jboss.web:name=http-0.0.0.0-8080,type=ThreadPool

See http://community.jboss.org/wiki/JBossMonitoring for more info.

Note

JBoss AS does not by default wait for the monitored objects to be initialized before the monitors are started. Since the monitoring service starts before the web engine, all monitors that depend on the web infrastructure or applications will simply fail to start. This issue is solved using the deployment dependencies, or by starting the monitors at runtime after the server has been fully initialized.
